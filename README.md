# ai-workspace

## 先抓住这个仓库的用途

`ai-workspace` 是一个专门面向 AI 的只读浏览仓库，用来集中存放多个项目的源码、文档和分析材料。它把本地分散的工程整理成统一入口，使  AI 能够浏览目录、读取文件、搜索符号、追踪跨文件引用，并沿着真实代码理解模块关系和执行流程。

仓库地址：https://github.com/pingzai-2/ai-workspace

可以把它理解成三种视角下的同一个工作区：

- 对本地而言，它是多个项目的统一汇总目录。
- 对 GitHub 而言，它是本地内容的远程镜像。
- 对 AI 而言，它是可以直接搜索和阅读的只读代码浏览器。

## 仓库边界

这里不是正式开发仓库，也不维护各子项目原有的 Git 历史。各项目内部的 `.git`、编译中间产物、缓存、临时文件和敏感配置由根目录 `.gitignore` 统一过滤，不上传到 GitHub。

重新拉取本仓库后，各项目只表现为普通目录，不包含原项目的分支、提交记录和远程配置。源码修改仍应回到原始开发仓库完成；这里主要承担汇总、远程浏览和 AI 解读入口的职责。

## 内容组织

```text
ai-workspace/
├── <project-a>/     某个独立项目、源码目录或资料集合
├── <project-b>/     另一个项目、版本或阅读对象
├── <archive>        按需保留的原始归档或附件
├── .gitignore       汇总仓库过滤规则
├── README.md        仓库入口与 AI 阅读规范
└── sync-project.sh  本地内容同步脚本
```

项目目录、资料和归档会随当前任务按需增加、移除或替换，上面的名称只表示组织方式，不是固定清单。开始分析时，应先读取实际目录并确认用户指定的对象；每个一级子目录默认视为独立阅读对象，不要把相邻项目、归档文件或历史材料自动混成同一个工程。

## AI 导读风格

### 先建立直觉，再进入细节

不要一开始就罗列目录、函数和行号。先用一小段话回答：当前材料在解决什么问题、主线从哪里开始、最终走向哪里。然后给出 2～5 步阅读路径，让读者先形成方向感。

推荐顺序：

```text
一句话用途判断
  -> 2～5 步阅读主线
  -> 3～7 个关键节点
  -> 节点之间的变化与职责交接
  -> 已确认范围和未展开部分
```

### 像导师带着读

每个重点段落围绕一个问题、阶段或职责交接展开，优先讲清：

- 为什么需要这一步。
- 进入前处于什么状态。
- 当前代码或材料改变了什么。
- 完成后控制权、数据或职责交给哪里。
- 哪些结论来自实际读取，哪些只是推断。

可以使用“这里先不要钻进……”“先记住这次交接……”这类引导，但必须准确。不要把解读写成文件清单、路径账单或“一行一个函数”的同构摘要。

### 把节点自然嵌入正文

目录、文件、单行和连续范围应当作为正在解释的节点自然出现在句子里，不要在文末另堆一组路径让读者自己对应。

label 以当前语境下最短且不歧义为准，可以使用项目相对路径、短文件名、符号名或自然说明。讲一个语句时使用单行，讲连续逻辑时使用范围；如果 label 包含 `:line` 或 `:start-end`，实际目标必须保留相同后缀。

```text
目录：linux-0.11-VC/boot/
文件：linux-0.11-VC/boot/head.s
单行：linux-0.11-VC/boot/head.s:22
范围：linux-0.11-VC/init/main.c:153-177
符号：setup_paging()
```

当前环境提供点击跳转能力时，使用它渲染已经确定的 label 和目标；没有专用能力时，使用原始 Markdown label。底层编码只负责打开正确位置，不得反过来改变正文风格、阅读顺序或节点选择。

### 尊重实际读取边界

只对本轮真正读过的文件和范围下结论。没有展开的目录明确标为“未展开”，证据不足的判断明确标为推断。目录规模或结构使浅层阅读无法可靠确定入口时，应先说明边界，再继续补充读取，不要把局部观察包装成完整结论。

## 本地维护与同步

仓库内容由本地手动维护，可以直接新增、修改或删除项目目录和文件。完成调整后执行：

```sh
./sync-project.sh
```

脚本依次执行 `git add -A`、创建提交并推送到远程。它不会修改或删除子项目源码，但会把工作区当前的新增、修改和删除统一记录到汇总仓库；运行前应先确认本轮变更范围。

## 工作流程

```text
本地维护 ai-workspace
        ↓
检查目录与文件范围
        ↓
执行 sync-project.sh
        ↓
提交并推送到 GitHub
        ↓
AI 从远程只读浏览、搜索和解读
```

阅读某个项目时，先从对应一级目录进入，再沿 README、入口文件、关键逻辑和测试或样例逐步展开。解释应始终服务于理解主线，而不是证明读取过多少文件。
